# v0.2.48 validation guide

Step-by-step checklist for proving the new `creator.RunCreate` pipeline
ships value on a real VM before flipping its env-var default to ON
in v0.2.49. Every step has an exact command to paste and an expected
output to match; if anything diverges, copy the divergence into a bug
report and revert with the one-liner at the bottom.

The whole exercise takes ~15 minutes on a fresh Debian 13 VM.

---

## Prerequisites

- A fresh Debian 13 (trixie) or Ubuntu 24.04 LTS host. amd64 or arm64.
- Root SSH access.
- A real domain you control DNS for, OR `--resolve` flag patience.
- Total tasks the VM will run: dpkg install, `auracp-install`, one
  panel admin signup, two site creates. ~5 GB disk, ~1 GB RAM during
  the install spike.

---

## 1. Install v0.2.48

```bash
# On the VM, as root:
ARCH=$(dpkg --print-architecture)
curl -fL -o /tmp/auracp.deb \
  "https://github.com/tapashdatta/auraCP/releases/download/v0.2.48/auracp_0.2.48_${ARCH}.deb"
dpkg -i /tmp/auracp.deb

# Provision the data plane non-interactively:
auracp-install --yes --db=both --node=yes --php=yes \
  --panel-domain=panel.example.test    # use a real domain you control
```

Expected: `auracp-install` finishes in 3-6 minutes. The control panel
should be reachable at `https://panel.example.test` (or
`https://<server-ip>:8443` if you skipped `--panel-domain`).

Verify daemon health:

```bash
systemctl status auracpd                  # active (running)
curl -kfsS https://127.0.0.1:8443/api/health
# {"status":"ok"}
```

Create your admin user via the panel signup screen.

## 2. Reproduce the legacy behaviour (control group)

Without flipping the feature flag — this is the baseline behaviour
v0.2.47 had. Create a PHP site through the panel: type=PHP, domain=
`legacy.example.test`, PHP version 8.3.

Verify the site landed:

```bash
ls /etc/nginx/sites-enabled/ | grep legacy
# legacy.example.test.conf

ls /etc/php/8.3/fpm/pool.d/ | grep legacy
# legacy.example.test.conf

systemctl is-active nginx php8.3-fpm
# active
# active
```

Site responds:

```bash
curl -kI --resolve legacy.example.test:443:127.0.0.1 https://legacy.example.test/
# HTTP/2 200
# server: nginx/1.30.x
```

Good — legacy path works. Delete this site through the panel before
proceeding.

---

## 3. Flip the new pipeline ON

```bash
systemctl set-environment AURACP_USE_NEW_CREATOR=1
systemctl restart auracpd

# Confirm the env var landed:
systemctl show auracpd -p Environment | grep AURACP_USE_NEW_CREATOR
# Environment=AURACP_USE_NEW_CREATOR=1
```

In one terminal, start tailing the structured slog. Leave this open
for the rest of the test:

```bash
journalctl -u auracpd -f -o cat | grep -E "step=|site="
```

---

## 4. Create a PHP site through the new pipeline

Through the panel: type=PHP, domain=`new.example.test`, PHP version 8.3.

In the journalctl tail, you should see something like:

```
step=CreateUser site=new.example.test took_ms=312
step=CreateRootDirectory site=new.example.test took_ms=4
step=CreateLogrotateFile site=new.example.test took_ms=2
step=CreateSslCertFiles site=new.example.test took_ms=178
step=CreateIndexPhp site=new.example.test took_ms=1
step=CreatePhpFpmPool site=new.example.test took_ms=43
step=CreateNginxVhost site=new.example.test took_ms=11
step=ReloadNginx site=new.example.test took_ms=87
step=ResetPermissions site=new.example.test took_ms=141
```

Total: under one second. If any line says `step failed`, copy that
line into a bug report and **stop here** — the new path has surfaced
something the legacy path was masking.

Verify the artifacts are consistent (this is the property test):

```bash
# Extract the user from the vhost:
VHOST_USER=$(grep -oP 'root /home/\K[^/]+' /etc/nginx/sites-enabled/new.example.test.conf)
# Extract the user from the FPM pool:
POOL_USER=$(grep -oP '^user = \K.+' /etc/php/8.3/fpm/pool.d/new.example.test.conf)

echo "vhost user: $VHOST_USER"
echo "pool  user: $POOL_USER"
test "$VHOST_USER" = "$POOL_USER" && echo "✓ MATCH" || echo "✗ DRIFT"
```

Expected: both report the SAME username, `✓ MATCH`. This is the
property the new pipeline structurally guarantees.

Smoke-test the site:

```bash
curl -kI --resolve new.example.test:443:127.0.0.1 https://new.example.test/
# HTTP/2 200
curl -ks --resolve new.example.test:443:127.0.0.1 https://new.example.test/ | head -5
# auraCP — new.example.test ready.
```

## 5. Reproduce the original a.garuda.sh bug ATTEMPT (negative test)

The whole point of v0.2.48 is that this scenario is now caught at
create time. Try to create a duplicate-domain site:

```bash
# In the panel, try to create another PHP site with domain new.example.test.
# The API should reject with HTTP 400 and a message containing:
#   "domain conflict: nginx vhost for new.example.test already exists"

# Or via the auracp CLI:
auracp sites:create --domain=new.example.test --type=php --php-version=8.3 \
  --user=newuser 2>&1
# Error: domain conflict: nginx vhost for new.example.test already exists at
#   /etc/nginx/sites-available/new.example.test.conf (delete the existing
#   site first, or rerun with conflict=force)
```

Now the deeper test — manually create a stale pool file simulating the
"deleted but not cleaned up" scenario that produced the a.garuda.sh
bug:

```bash
# 1. Delete the site cleanly through the panel — vhost + pool + user all gone.
# 2. Simulate a stale orphan pool from a hypothetical PHP version switch
#    that the legacy path forgot to clean up:
mkdir -p /etc/php/8.4/fpm/pool.d
cp /tmp/.auracp-test-orphan.conf 2>/dev/null || cat > /etc/php/8.4/fpm/pool.d/new.example.test.conf <<EOF
[new.example.test]
user = old-orphan-user
group = old-orphan-user
listen = /run/php-fpm/new.example.test.sock
pm = ondemand
pm.max_children = 5
EOF

# 3. Try to create new.example.test again. Preflight should catch the
#    orphan and refuse:
auracp sites:create --domain=new.example.test --type=php --php-version=8.3 \
  --user=freshuser 2>&1
# Error: domain conflict: stale FPM pool exists at
#   /etc/php/8.4/fpm/pool.d/new.example.test.conf (orphan from a previous
#   create that didn't clean up — remove the file manually or
#   delete-then-recreate the site)

# 4. Clean up the orphan + retry — should succeed:
rm /etc/php/8.4/fpm/pool.d/new.example.test.conf
# (re-run the create through the panel — should land cleanly)
```

If both rejections above fired, Preflight is doing its job — the bug
class is structurally extinct.

## 6. Smoke-probe failure path

Force a smoke-probe failure to verify the new pipeline catches an
empty-body symptom at create time. We sabotage a vhost mid-create:

```bash
# Skip this if you don't have time — it's a destructive sanity check.
# Easiest: create a site through the panel, then while it's mid-create,
# corrupt its docroot perms so PHP-FPM can't read the index:
chmod 000 /home/<the-new-user>/htdocs/<domain>

# The API response should include the smoke-probe error, citing the
# vhost↔pool diagnostic. Don't just trust the curl output — confirm
# in journalctl:
journalctl -u auracpd | grep smoke
# smoke probe failed: HTTP 200 returned an empty body
#   (vhost↔pool user mismatch is the most likely cause; check
#   /etc/nginx/sites-enabled/<domain>.conf vs /etc/php/*/fpm/pool.d/<domain>.conf)
```

Restore perms once you've seen the error:

```bash
chmod 755 /home/<the-new-user>/htdocs/<domain>
```

## 7. Cross-PHP-version delete sweep

The other half of the structural fix. Simulate the "site uses 8.3 then
switches to 8.4 then deletes" sequence:

```bash
# 1. Create a PHP site, version 8.3:
#    (through the panel — name it `sweep.example.test`)
# 2. Confirm 8.3 pool exists:
ls /etc/php/8.3/fpm/pool.d/sweep.example.test.conf
# /etc/php/8.3/fpm/pool.d/sweep.example.test.conf

# 3. Switch to PHP 8.4 through the panel (Settings → PHP runtime → 8.4 → Apply)
ls /etc/php/8.4/fpm/pool.d/sweep.example.test.conf
# /etc/php/8.4/fpm/pool.d/sweep.example.test.conf
ls /etc/php/8.3/fpm/pool.d/sweep.example.test.conf 2>&1
# (should be gone — v0.2.47 introduced this sweep; v0.2.48 keeps it)

# 4. Delete the site through the panel. Now verify NO pool survives in ANY version:
find /etc/php -path '*/fpm/pool.d/sweep.example.test.conf' 2>/dev/null
# (no output — the v0.2.48 RunDelete swept every installed version)

# 5. Recreate the site. It should land cleanly because Preflight saw no orphan.
```

## 8. Performance sanity

The new pipeline is supposed to be SNAPPIER than the legacy one. Time
five back-to-back site creates and report the median:

```bash
# Through the CLI, time five creates:
for i in 1 2 3 4 5; do
  time auracp sites:create --domain=perf$i.example.test --type=php \
    --php-version=8.3 --user=perf$i
done 2>&1 | grep real

# Then delete them all and run the same against the legacy path (env var off):
for i in 1 2 3 4 5; do auracp sites:delete perf$i.example.test; done
systemctl unset-environment AURACP_USE_NEW_CREATOR
systemctl restart auracpd
# repeat the timing loop

# Re-enable the flag when done.
```

Expected: new path is ~10-15% faster (single nginx reload vs 2-3).
Same order of magnitude — under 2 seconds per site on a 1-vCPU VM.

---

## 9. Instant revert

If anything misbehaves at any point during the validation, this is the
single-command revert. No rebuild, no deb reinstall:

```bash
systemctl unset-environment AURACP_USE_NEW_CREATOR
systemctl restart auracpd
# Done. The legacy site.Manager.Create path is back in effect.
```

Existing sites created via either path keep working — the on-disk
shape is identical between the two.
